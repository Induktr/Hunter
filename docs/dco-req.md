# Спецификация Требований (SRS)

### 1. Функциональные Требования (FR)

#### Модуль 1: "Уши" (Listener)
*   **FR-1.1:** Система должна подключаться к Telegram как Userbot (MTProto) и слушать события `NewMessage`.
*   **FR-1.2:** Система должна игнорировать сообщения из каналов, не внесенных в "Белый список" (`config/channels.json`).

#### Модуль 2: Фильтрация (Noise Reduction)
*   **FR-2.1:** Система должна мгновенно (без ИИ) отбрасывать посты, содержащие стоп-слова из "Черного списка" (Sales, Gambling, Office, etc.).
*   **FR-2.2:** Система должна пропускать дальше только посты, содержащие хотя бы одно ключевое слово (React, Frontend, JS, etc.).

#### Модуль 3: "Мозг" (AI Analysis)
*   **FR-3.1:** Отфильтрованный текст отправляется в Gemini 2.0 Flash Lite.
*   **FR-3.2:** ИИ должен вернуть JSON с полями: `score` (1-10), `summary`, `cover_letter`.
*   **FR-3.3:** ИИ должен использовать методологию Криса Восса (Labeling, Calibrated Questions) при генерации отклика.

#### Модуль 4: Уведомления (Notification)
*   **FR-4.1:** Если `score` >= 7, система отправляет уведомление в личного бота.
*   **FR-4.2:** Уведомление должно содержать: Оценку, Название компании, Зарплату, Ссылку и Сгенерированный отклик.

### 2. Нефункциональные Требования (NFR)

*   **NFR-1 (Latency):** Время от публикации поста в канале до уведомления на телефоне — не более 10 секунд.
*   **NFR-2 (Reliability):** Автоматический перезапуск при разрыве соединения с Telegram (Reconnect).
*   **NFR-3 (Cost):** Использование бесплатных лимитов Gemini (или минимальных платных).

### 3. Ограничения (Constraints)
*   Работа на слабом железе (Dell 5290).
*   Соблюдение лимитов Telegram API во избежание бана аккаунта.
